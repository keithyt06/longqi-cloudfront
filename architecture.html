2 DATA FLOW

数据流架构

三条核心数据流：请求从用户到源站的完整路径、CloudFront 缓存策略决策树、以及基于标签的精准缓存失效机制。

↗

请求流 Request Flow

1
DNS 解析
Route 53 ALIAS → CloudFront Edge (最近 PoP)
2
WAF 规则评估
6 条规则依优先级逐条检查，Block/Count/Challenge
3
CF Function (Viewer Request)
URL 重写 / A/B 分流 / 地理重定向
4
Cache 检查
命中 → 直接返回；未命中 → 回源
5
Origin (S3 / ALB)
S3 OAC 签名 / VPC Origin ENI → Internal ALB
6
Response Headers Policy
注入安全头 HSTS/X-Frame/CSP，缓存并返回
💾

缓存流 Cache Policy

  • 24h CachingOptimized — /static/* /images/*
    静态资源，Brotli + Gzip 压缩
  • 1h ProductCache — /api/products*
    按 page/category/sort QS 区分缓存键
  • 60s PageCache — Default (*)
    SSR 页面，仅转发 x-trace-id + waf-token
  • 0 CachingDisabled — /api/cart /api/user /api/debug
    全部透传，不缓存
Cache Key 组成：URL 路径 + 转发的 Query String + 白名单 Cookie + 压缩格式。精准控制缓存键粒度是 CloudFront 性能优化的关键。
🏷

失效流 Tag Invalidation

1
Origin Response + Cache-Tag
Express 在响应头添加 Cache-Tag: product-123, category-wigs
2
Lambda@Edge 提取标签
解析 Cache-Tag → SNS 发布 → SQS 聚合
3
DynamoDB 存储映射
PK=tag SK=url，TTL 跟随 Cache-Control max-age
4
Admin API 触发失效
POST /api/admin/invalidate-tag { tags: [...] }
5
Step Functions 编排
查询 DynamoDB → URL 去重分组 → 速率控制 → CreateInvalidation

VPC Origin 架构 (本平台)

  • ALB 零公网暴露，Internal scheme
  • 4 层安全组链式信任，最小权限
  • 流量走 AWS 骨干网，延迟更低
  • 无需 NAT Gateway 的源站访问路径
  • DynamoDB 通过 VPC Endpoint (Gateway) 访问

传统 Public ALB 架构

  • ALB 暴露公网 IP，可被直接访问
  • 绕过 CloudFront/WAF 直攻源站
  • 需额外配置 IP 白名单 (CloudFront IP 范围经常变更)
  • 无法利用 AWS 骨干网加速
  • 安全组配置复杂，容易遗漏
4 UUID TRACKING

UUID 链路追踪设计

服务端 UUID 方案：EC2 Express 中间件生成 UUID v4，通过 HttpOnly Cookie 持久化（2 年）。登录后绑定 Cognito 账号，实现跨设备统一追踪。广告拦截器无法屏蔽。

👤 匿名用户流程
1
首次访问
请求进入 Express，uuid-tracker 中间件检查 Cookie
2
生成 UUID v4
无 Cookie → 生成新 UUID，设置 req.traceId
3
Set-Cookie 返回
x-trace-id=UUID; HttpOnly; Secure; SameSite=Lax; Max-Age=63072000
4
后续请求
自动携带 Cookie，中间件读取 UUID 设置 X-Trace-Id 响应头
🔒 登录用户流程
1
Cognito 登录
用户名/密码 → Cognito API → JWT (access_token + id_token)
2
查询 DynamoDB
PK = cognito_user_id 查找已绑定的 UUID
3
UUID 统一
存在 → 覆盖当前 Cookie 为数据库 UUID (跨设备统一)
不存在 → 绑定当前 UUID 到 DynamoDB
4
返回 JWT + UUID
Set-Cookie 更新 UUID，后续所有设备登录后使用统一 trace_id
📊 DynamoDB: unice-trace-mapping
属性 类型 角色 说明
cognito_user_id String PK Cognito 用户唯一标识符 (sub)
trace_id String GSI-PK UUID v4，GSI: trace_id-index 支持反查
created_at String - 首次绑定时间，ISO 8601
last_device String - 最近登录设备 User-Agent 摘要
last_seen String - 最近活跃时间，ISO 8601
匿名用户特性
UUID 仅存于 Cookie（2 年有效），清 Cookie 即丢失。每设备独立，不写入 DynamoDB。HttpOnly 属性防止前端 JS 篡改。
登录用户特性
UUID 永久绑定 Cognito 账号，DynamoDB 记录。任何设备登录后统一为同一 UUID，支持跨设备归因分析。
5 SECURITY

WAF + CloudFront 安全体系

多层纵深防御：WAF 6 条规则链覆盖 OWASP Top 10，Bot Control Targeted 级别 + JS SDK 浏览器指纹识别，CloudFront 原生安全功能矩阵提供额外保护。

WAF 规则优先级链

Priority 规则名称 类型 动作 检测原理
1 AWS Common Rule Set AWS 托管 COUNT 正则匹配 SQLi/XSS/SSRF/路径遍历等 OWASP Top 10 攻击模式，建议观察后切 Block
2 AWS Known Bad Inputs AWS 托管 BLOCK 检测已知恶意 payload，包括 Log4j JNDI (CVE-2021-44228)、Spring4Shell 等零日漏洞利用
3 AWS Bot Control (Targeted) AWS 托管 CHALLENGE 行为分析 + ML 模型，JS SDK 浏览器指纹验证，区分真实用户与自动化工具
4 RateLimitRule 自定义 BLOCK 同一 IP 5 分钟内 > 2000 次请求触发封禁，防暴力破解和简单 DDoS
5 AWS IP Reputation List AWS 托管 COUNT AWS 维护的恶意 IP 信誉库，标记僵尸网络节点、匿名代理、Tor 出口节点
6 GeoBlockRule 自定义 BLOCK 基于 MaxMind GeoIP 数据库识别请求来源国家，当前封禁 KP (朝鲜) / IR (伊朗)
